### MITSUBA AGENT 3 — Differentiable Rendering for Lens Design
**Priority: HIGH — this could replace the Julia solver entirely.**

**Task:** Research Mitsuba 3's differentiable rendering capability as a lens design tool.

**Background:**
Our current pipeline:
  1. Target image → Julia SOR (Optimal Transport) solver → OBJ mesh
  2. OBJ mesh → Python forward ray tracer → caustic verification image

Proposed alternative:
  1. Target image → Mitsuba 3 differentiable render → optimized OBJ mesh (directly)

**Questions to research:**
1. Can Mitsuba 3 (`llvm_ad_rgb` or `cuda_ad_rgb` variant) compute gradients of
   rendered caustic intensity with respect to surface vertex positions?
   — This requires the mesh to be a differentiable parameter.
2. What is the "caustic design by differentiable rendering" approach?
   Key papers to find:
   - "Differentiable Caustics" — any SIGGRAPH/ICCV/CVPR paper 2020-2024
   - Mitsuba 3 paper (Jakob et al. 2022) — does it demonstrate caustic design?
   - "Inverse Caustic Design" approaches
3. In Mitsuba 3, vertex positions can be parameters in `drjit`:
   ```python
   params = mi.traverse(scene)
   # Can we differentiate w.r.t. params['mesh.vertex_positions']?
   ```
4. What loss function connects rendered caustic → target image?
   - L2 pixel loss: |render - target|² — simple but may have many local minima
   - Optimal Transport loss: matches intensity distributions (same math as Julia!)
   - Perceptual loss: VGG features
5. What optimizer works for mesh vertex positions?
   - Adam with learning rate ~1e-4 for vertex coordinates
   - How many iterations to converge? (Likely 100-500 for coarse result)
6. Is the Mitsuba differentiable approach faster or slower than the Julia SOR solver
   for equivalent quality? (Julia SOR: ~45 min at 1024px; Mitsuba grad: unknown)
7. Are there existing examples of using Mitsuba 3 to design refractive optics?
   Search: "mitsuba3 inverse rendering lens" "differentiable caustic optimization"

**Deliverable:**

- If yes: outline of the optimization loop (loss function, optimizer, convergence criteria)
- If yes: does the output naturally produce an OBJ we can CNC mill?

## LUXCORE FIX AGENTS (2 agents — parallel with Mitsuba)

### LUXCORE AGENT 1 — clGetDeviceIDs Stub
**Priority: MEDIUM — if it works, gives us BIDIR which is the best quality.**

**New constraint discovered:** `clGetDeviceIDs` hangs from standalone Python main
thread, not just Blender background. `clGetPlatformIDs` works. SIP is enabled.

**Task:** Design and implement a clGetDeviceIDs stub that:
1. Returns `CL_DEVICE_NOT_FOUND` immediately (no IPC call)
2. Can be injected into the pyluxcore process

**Investigation paths:**
1. **DYLD_INSERT_LIBRARIES on Blender Python (not Blender binary):**
   Blender's Python (`python3.11` binary inside the app bundle) may not be
   code-signed/hardened in the same way as the main Blender binary.
   Test: `codesign -d --entitlements - /Applications/Blender.app/Contents/Resources/4.3/python/bin/python3.11`
   If NOT hardened → DYLD_INSERT_LIBRARIES works → write stub → run pyluxcore standalone.

2. **Stub C code (to write):**
   ```c
   #include <OpenCL/opencl.h>
   // Override only clGetDeviceIDs — let everything else pass through
   cl_int clGetDeviceIDs(cl_platform_id platform, cl_device_type device_type,
                          cl_uint num_entries, cl_device_id *devices,
                          cl_uint *num_devices) {
       if (num_devices) *num_devices = 0;
       return CL_DEVICE_NOT_FOUND;
   }
   ```
   Compile: `cc -dynamiclib -o /tmp/fake_cl_devices.dylib fake_cl.c`
   Use: `DYLD_INSERT_LIBRARIES=/tmp/fake_cl_devices.dylib python3.11 script.py`

3. **Alternative: ctypes monkey-patch at Python level:**
   Before importing pyluxcore, override `clGetDeviceIDs` in the OpenCL dylib
   using ctypes to replace the function pointer. This avoids DYLD entirely.
   ```python
   import ctypes
   lib = ctypes.CDLL('/System/Library/Frameworks/OpenCL.framework/OpenCL')
   # Replace clGetDeviceIDs with a no-op
   # Challenge: pyluxcore loads OpenCL directly in C++, ctypes can't intercept that
   ```
   Note: This won't work if pyluxcore loads OpenCL via `dlopen` before Python
   has a chance to patch it.

4. **Build LuxCore from source without OpenCL:**
   The LuxCore CMake option `LUXRAYS_DISABLE_OPENCL=ON` produces a CPU-only build.
   This completely removes the `AddDeviceDescs()` call.
   Sources: https://github.com/LuxCoreRender/LuxCore
   Build complexity: HIGH (requires Embree, OIDN, OSL, OIIO, Boost)
   Time estimate: 2-4 hours if dependencies cooperate on macOS.

**Deliverable:** Working pyluxcore with LuxCore BIDIR/PATHCPU running, OR a clear verdict
on which approach is viable and step-by-step instructions for Claude Code.
